# 风格驱动 PPT 工作流

## 最高优先级硬规则：只允许 Image 2 出图

本 Skill 的所有最终幻灯片图片、缩略图板、视觉页、信息图、文章配图和 PPT 页面图片，必须使用 Image 2 模型生成。

严禁使用 HTML 输出图片。不得创建 HTML/CSS 页面、浏览器截图、SVG/HTML mockup、canvas render、PIL/脚本仿制图或任何本地渲染结果来作为最终或中间图片输出。Image 2 不可用时，不得改用 HTML 兜底；必须停止并说明需要 Image 2 访问权限。

这条规则优先级高于本文档内所有其它流程、快捷方式、交付规则或实现便利。PPTX 只能在 Image 2 生成并经用户确认页面图片之后，作为图片容器组装。

## 1. 风格提炼

从用户提供的图片、PPT、网页、海报、截图或文字描述中提炼可复用风格资产。输出时覆盖：

- 风格定位：适合什么内容、气质是什么、不适合什么。
- 颜色系统：背景色、主色、辅色、文字色、强调色。
- 字体气质：标题、正文、数字、标签的感觉。
- 构图语言：封面、内容页、比较页、流程页、结论页的排版倾向。
- 图形元素：线框、卡片、图标、纹理、照片、插画、材质、光影。
- 信息密度：每页/每图适合多少文字、多少模块。
- PPT 适配：哪些元素适合直接生成在整页图片中，哪些页面需要更少文字以保证 Image2 可读性。
- 提示词骨架：可替换主题、标题、模块、视觉锚点、比例。
- 负面约束：明确禁止的风格偏移。
- 风格隔离：明确本风格不能混入哪些之前见过的风格、材质、构图或装饰语言。

提炼后询问用户是否保存。保存时写入 `styles/{style-name}.md`，并给出可调用别名。

保存后，下次开始做 PPT 时先询问或判断：

- 是否调用已有风格模板。
- 是否基于用户提供的新图片提取风格 DNA 并生成新模板。
- 是否从默认风格开始。

## 2. 风格资产库

风格文件是可替换资产，不是 Skill 本体。每个风格文件应包含：

- 风格名称与别名。
- 适用场景。
- 视觉 DNA。
- Style Lock：该风格专属的跨页控制系统。
- 色彩、字体、构图、图形元素。
- PPT 页面类型适配。
- 图片/配图提示词骨架。
- 负面约束。
- 质量检查清单。

当用户说"调用某个风格"时，先在 `styles/` 中匹配文件名、中文名、别名或明显语义。

生成多页图片版 PPT 时，必须先从所选风格文件中提取 `Style Lock`，并把它作为每一页 Image2 prompt 的共享前缀。不要使用通用 Style Lock 替代风格专属 Style Lock。

风格隔离是硬规则：一套 PPT 只能使用一个已选 style source 和一个 Style Lock。不要把当前网页/图片提取出的风格，与之前任务里的古典、终端、海报、拼贴、杂志或其它视觉 DNA 混合，除非用户明确要求 hybrid style。

## 3. 文档理解

读取用户提供的 Markdown、TXT、PDF、Word、PPT 或大纲路径后，先做内容结构化：

- 主题与一句话主张。
- 目标受众与使用场景。
- 3 到 6 个核心观点。
- 可以视觉化的对象：流程、对比、框架、时间线、清单、案例、数据。
- 建议页数或图片数量。
- 每页/每图的标题与主要信息。

长文档不要照搬。每页只表达一个主判断，正文压缩成短句、标签、数据或关系图。

默认语言规则：以中文为主，尤其是页面标题、章节标题、模块标题、结论句和说明文字。所有输出文档也以中文为主，包括 `outline.md`、`prompts.md`、`style-used.md`、`revision-log.md` 和最终交付说明。英文只保留在产品名、模型名、品牌名、代码/API 术语、专有名词、源文引用或翻译后更不清晰的领域词里，例如 `Claude`、`Image2`、`API`、`Opus 4.6`、XML 标签名。双语输出时也优先中文在前、英文为辅。

## 4. 生成前参数确认

如果用户未提供且会影响结果，开始生成前确认：

- 输出类型：图片序列、图片版 PPTX，或两者都要。默认先生成图片序列；用户说"做 PPT"时默认再组装图片版 PPTX。
- 页数：直接询问用户需要多少页；若用户让系统决定，则根据文档长度建议页数。
- 比例：默认 16:9；可选 4:3、1:1、4:5、3:4、9:16。
- 尺寸与质量：默认 2K。
- 语言：默认中文优先，尤其标题和输出文档正文必须优先中文；除非用户明确要求英文或双语，不需要追问语言。
- 文字密度：默认低密度。普通 PPT 内页只放 1 个标题、1 个短副标题或结论、最多 2-3 个短信息点；除非用户明确要求留存阅读型报告或信息图，不允许密集文字。
- 是否放日期、作者、品牌名、水印。Logo/固定字标默认不交给 Image2 生成；需要 Logo 时建议后期用固定图片或文字层添加。
- 是否需要先输出大纲/提示词再生成。
- 是否需要同时交付 `prompts.md` 和 `outline.md`。

不要机械追问已明确的信息。能从上下文安全推断的，直接采用并在输出计划中说明。

多页 deck 的硬规则：生成图片前必须先输出 `outline.md` 和 `prompts.md` 并等待用户确认，除非用户明确说跳过确认直接生成。`prompts.md` 必须包含完整 Style Lock，不要只给单页提示词。

## 5. 日期与时间

只要画面或 PPT 中出现日期、时间、今天、日报、周报、月报、版本日、日志日、发布时间或时间戳，必须先确认日期。

优先级：

1. 用户明确提供的日期。
2. 当前会话环境日期。
3. 本地系统时间或可靠时间工具。
4. 如果无法确认，向用户询问。

未确认前不要把日期写入图片提示词或 PPT 页面。

## 6. Outline / Prompt / Image2 路线

标准顺序：

1. 选择或提取风格。
2. 生成 `outline.md`：每页页型、页面角色、标题、副标题、信息块、视觉锚点。
3. 生成 `prompts.md`：完整 Style Lock + 每页 Image2 prompt。
4. 停下来让用户确认。
5. 用户确认后再调用 Image2 生成图片。
6. 生成图片后，先展示或列出所有图片，让用户确认全部图片是否通过。
7. 如果用户要求修改，按页重生成或替换图片，直到用户确认全部图片。
8. 用户确认全部图片后，把图片全屏铺入 `deck.pptx`。
9. 把所有最终图片、图片版 `deck.pptx`、`prompts.md`、`outline.md`、`style-used.md` 和修改记录打成 `.zip`。
10. 将 `.zip` 存到用户默认保存文件夹；macOS 默认使用 `~/Downloads`，除非用户指定了其它默认保存位置或环境中有明确配置。
11. 生成后询问是否保存本次风格模板，方便下次直接调用。

`outline.md` 和 `prompts.md` 的文档正文与可见文案默认使用中文。标题不应直接照搬英文原文，除非它是品牌、模型、产品或必须保留的专有名词；可以采用“中文标题 + 英文术语”的形式，例如“选择合适的 Claude 模型”。提示词文档中的风格说明、页面说明、质量约束、负面约束也应中文为主，必要英文短语只作为模型更容易理解的补充。

先判断用户要哪种交付：

- 图片序列：每页使用 Image2 生成一张完整幻灯片图片，适合信息图、文章配图、小红书图和视觉稿。
- 图片版 PPTX：先生成图片序列，再把每张图片按 16:9 全屏铺入 PPTX，适合用户需要 PPT 文件但不需要可编辑对象。
- 完整输出包：用户确认全部图片后，交付一个 `.zip`，内含图片序列、图片版 PPTX、每页提示词、页面大纲和使用的风格文件副本。

用户只说"做 PPT"时，默认走图片版 PPTX，不追问是否可编辑。用户只说"做几张图"或"信息图"时，输出图片序列即可；只有用户要 PPT 文件时才组装为 PPTX。无论哪种多页输出，最终打包前都必须等用户确认全部图片通过。

## 7. Image2 制作规则

图片页输出应优先保持视觉效果、风格一致和文字可读：

- 页面类型必须从 `page-types.md` 中选择或组合：封面、目录、核心观点、三栏对比、双栏对照、流程、框架、时间线、数据、案例、清单、结论。
- 每页 1 个核心判断。普通内页最多 2-3 个短信息点；只有用户明确要信息图/报告页时才允许 4-5 个信息点。
- 默认中文可见文案；标题、模块名、结论句优先中文，必要英文术语放在中文语境里。
- 封面页可以更花哨、更沉浸、更像视觉海报；内页必须回到 PPT 初衷：文字和信息呈现优先，图像、背景和装饰只做辅助。
- 内页不要做成连续海报或插画集。背景应低对比、低复杂度，装饰元素减少，给标题和信息模块留出清晰阅读区。
- 默认不做信息密集页。除非用户明确要求信息图或留存阅读材料，内页每页最多 2-3 个主要信息块，每块只放 1 行短句；不要同时放长副标题、长解释、多个标签和脚注。
- 不把长段落塞进页面；把文章语言改写成演示语言。
- 每页提示词必须明确：比例、页面类型、标题、副标题、模块文字、视觉锚点、风格 DNA、颜色、负面约束。
- 每页提示词必须继承同一个风格专属 Style Lock，用来固定字体气质、字号层级、版式网格、装饰小元素、色彩比例、页脚和禁用项。
- 每页提示词必须继承同一套边框/容器系统。不要每页换一种卡片、框架、线条、圆角、阴影、箭头或装饰边；统一组件比局部花样更重要。
- 默认不要让 Image2 生成 Logo、品牌字标、重复页眉字标、水印或跨页固定英文标识。模型生成的这类小文字很容易跨页漂移；如用户需要 Logo，只在画面中预留干净位置，最终由用户或后期固定资产层添加。
- 每页提示词必须写清“不要混入其它已见风格”，特别是不要混入当前任务之前出现过的参考风格、生成尝试或历史 deck 的视觉元素。
- 每页提示词必须声明页面角色：`cover` 或 `inner slide`。`inner slide` prompt 必须包含“information-first, quiet background, restrained ornaments, readable text hierarchy”或等价中文约束。
- Style Lock 要尽量具体：主色、辅色、背景色、文字色、边框色、强调色都要给出色号；字体要给出明确气质或候选字体名；边框要说明直角/圆角、线宽、细线/双线/几何线框/玻璃拟态等形态。
- Image 2 生成整页成品图；严禁使用 HTML、PIL、SVG、canvas、本地脚本或浏览器截图仿制最终视觉稿或中间视觉稿。
- 如果中文文字较多，先压缩内容或拆页，不要把密集段落交给 Image2。

## 8. 图片与配图规则

默认 16:9、Image2、高分辨率。多图输出时：

- 保持同一个风格文件的视觉 DNA。
- 每张图使用不同页面类型：封面型、框架型、对比型、流程型、结论型。
- 每张图默认不要超过 3 个信息区；只有明确的信息图需求才可到 5 个。
- 中文短句优先，避免小字和密集段落。
- 明确画面比例、主标题、副标题、模块、视觉锚点、负面约束。

## 9. 后期修改

用户对其中一张图、一页 PPT 或某个局部不满意时，进入 `revision-workflow.md`：

- 先确认修改对象：第几页/第几张图。
- 再确认修改方向：文字、构图、颜色、风格、视觉主体、页型、日期、Logo 或图表。Logo 相关修改优先走后期固定资产层，不优先重生成整页。
- 默认只修改对应页/图，不重做整套。
- 图片版 PPTX 要同步替换该页图片并重新导出。
- 不处理可编辑 PPTX 对象；如果用户要改文字、构图、配色或视觉主体，重写该页 Image2 提示词并重生成该页图片。
- 修改后记录版本或在最终回复中说明修改范围。

## 10. 交付组织

复杂任务建议使用如下目录：

```text
outputs/YYYY-MM-DD-project-title/
├── deck.pptx
├── YYYY-MM-DD-project-title.zip
├── images/
├── versions/
├── prompts.md
├── style-used.md
├── revision-log.md
└── outline.md
```

图片生成后先让用户确认。用户确认全部图片通过后，最终交付应创建 `.zip` 并保存到用户默认保存文件夹，压缩包内必须包含最终图片、图片组成的 `deck.pptx`、提示词文档和大纲文档。最终回复给出 `.zip` 文件链接，并简述页数和风格。

详见 `output-package.md`。

## 11. 质量检查

交付前检查：

- 风格是否与选定 style 文件一致。
- 页面/图片是否符合比例与质量要求。
- 页数是否符合用户确认。
- 每页是否已匹配页面类型。
- 中文是否清晰、短、可读。
- 标题是否一眼能读懂。
- 模块是否对齐且不超过信息密度。
- 是否一眼就知道先看哪里；如果出现多个同等重量的框、标题、标签、脚注抢焦点，必须重写 prompt 降低文字量和视觉重量。
- 边框、容器、分割线、箭头、圆角/直角是否全套统一；如果每页框架形态不同，必须重写 prompt 统一组件系统。
- 是否混入了上一套风格或历史参考图的视觉 DNA；如果混入，必须重写 prompt 强化风格隔离。
- 是否出现模型生成的 Logo/品牌字标/固定页眉标识漂移；如果出现，删除这些元素或改为后期固定资产层。
- 内页是否信息优先，而不是被背景图、装饰元素或花哨场景抢走注意力。
- 是否存在每页都像封面海报的问题；如果存在，重写内页 prompt 降低视觉复杂度。
- 日期是否已经确认。
- PPTX 是否只是图片容器，且每页图片已全屏铺满。
- 最终 `.zip` 是否在用户默认保存文件夹中，且包含全部图片、图片版 PPTX、`prompts.md`、`outline.md` 和相关风格/修改记录文档。
- 多张图或多页 PPT 是否统一但不重复。
